# Core Concepts

This page explains the four building blocks of a Localbox solution and how they fit together.

---

## The big picture

```
solution.py
    │
    ├── SolutionConfig          global settings (name, branch, Docker project/network, env)
    │
    ├── Projects ─────────────► Git repos, cloned into .build/projects/
    │       │
    │       └── each Project has a Builder ──► Docker image + command that compiles sources
    │                                          Output: compiled artifact in .build/projects/<name>/
    │
    └── Services ─────────────► Docker containers orchestrated by Compose
            │
            └── each Service has a DockerImage ──► either a pull (postgres:16)
                                                   or a Dockerfile build (FROM jdk + COPY artifact)
```

`localbox compose generate` wires everything into `docker-compose.yml`.
`docker compose up -d` starts the containers.

---

## Solution

A **solution** is a directory that contains `solution.py`. That file is the entry point — Localbox discovers it by walking up the directory tree from wherever you run the command, the same way `git` finds `.git`.

```
my-solution/            ← solution root
├── solution.py         ← marks the root, defines everything
├── solution-override.py  ← per-developer secrets, gitignored
├── docker-compose.yml  ← generated by localbox
└── .build/             ← generated artifacts
    ├── projects/       ← cloned repos live here
    │   └── api/
    ├── logs/           ← build logs per project
    └── maven/          ← build cache (maven, gradle, node)
```

Everything in Localbox is relative to the solution root. When you run `localbox projects clone`, it clones into `<solution-root>/.build/projects/`. The `docker-compose.yml` is written to `<solution-root>/`.

### Multi-file solutions

For large solutions, split definitions across files:

```
my-solution/
├── solution.py         ← imports the sub-modules
├── projects/
│   ├── __init__.py     ← must exist for auto-import
│   ├── backend.py      ← JavaProject definitions
│   └── frontend.py     ← NodeProject definitions
└── services/
    ├── __init__.py
    ├── databases.py
    └── backend.py
```

Files in `projects/` and `services/` are auto-imported when they have an `__init__.py`. Objects defined there are collected automatically — no explicit import into `solution.py` is required.

## Projects

A **Project** represents an external Git repository. Localbox clones it, optionally applies patches, and then uses a Builder to compile it inside Docker.

```python
from localbox.models import JavaProject, NodeProject, Project, maven, gradle, node

# Java project
api = JavaProject(
    "backend:api",                         # name (group:local_name)
    repo="git@github.com:org/api.git",     # git URL
    jdk=17,                                # JDK version
    builder=maven(),                       # how to build it
    branch="main",                         # optional branch override
)

# Node.js project (builder defaults to node(20))
ui = NodeProject(
    "frontend:ui",
    repo="git@github.com:org/ui.git",
)

# Project with dependencies (built after its deps)
app = JavaProject(
    "backend:app",
    repo="git@github.com:org/app.git",
    jdk=17,
    builder=maven(),
    deps=[api],                            # will be built after api
)
```

### Name and grouping

The name follows the pattern `group:local_name`. A colon splits a project into a group and a local identifier:

Project targets under the `projects` domain are short-form: `<name>`, `<group>`, or `<group>:<name>` (no domain prefix).

| Name | Group | Local name | Target under `localbox projects …` |
|------|-------|-----------|------------------------------------|
| `"api"` | — | `api` | `api` |
| `"backend:api"` | `backend` | `api` | `backend:api` (or `backend` for the whole group) |
| `"libs:utils"` | `libs` | `utils` | `libs:utils` (or `libs` for the whole group) |

`localbox projects list` shows the tree. `localbox projects build backend` builds everything in the `backend` group.

### Clone directory

Each project is cloned into `.build/projects/<local_name>/`. For `"backend:api"` the clone goes to `.build/projects/api/`.

### Build order (dependencies)

Use `deps=` to declare build-time dependencies. Localbox computes topological order automatically:

```python
utils   = JavaProject("libs:utils", repo="...", jdk=8, builder=mvn)
core    = JavaProject("libs:core",  repo="...", jdk=8, builder=mvn, deps=[utils])
service = JavaProject("backend:svc",repo="...", jdk=8, builder=mvn, deps=[core])
# build order: utils → core → service
```

### Patches

Place `.patch` files in `patches/<local_name>/` and they are applied automatically after each clone in alphabetical order:

```
patches/
└── api/
    ├── 0001-fix-docker-startup.patch
    └── 0002-add-healthcheck-endpoint.patch
```

## Builders

A **Builder** defines the Docker image and command used to compile a project. The build runs entirely inside Docker — the developer's machine needs no JDK, Maven, Gradle, or Node.js installed.

```
.build/projects/api/   (host)
        │
        │  mounted at /var/src (inside container)
        ▼
┌───────────────────────────────┐
│  maven:3.9-amazoncorretto-17  │
│  $ mvn install -Dmaven.test.skip=true
│  /var/src/target/*.jar  ◄────── artifact lives here after build
└───────────────────────────────┘
        │
        │  .build/maven/  ◄──── cache volume, persisted across builds
```

### Pre-configured builders

```python
from localbox.models import maven, gradle, node

maven()         # Maven 3.9, JDK from project
maven("4.0")    # Maven 4.0
gradle()        # Gradle 8.14, JDK from project
gradle("9.0")   # Gradle 9.0
node(20)        # Node.js 20
node(22)        # Node.js 22
```

The **JDK version is a project property**, not a builder property. This is intentional: the same `maven()` builder works across all JDK versions. The Docker image is resolved at build time by combining builder version with the project's JDK:

```
maven("3.9") + jdk=17  →  maven:3.9-amazoncorretto-17
gradle("8.14") + jdk=21 →  gradle:8.14-jdk21
```

### JDK providers

By default `jdk=17` means Amazon Corretto 17. Use factory functions to select a different distribution:

```python
from localbox.models import corretto, temurin, graalvm

jdk=corretto(17)  # amazon corretto  → maven:3.9-amazoncorretto-17
jdk=temurin(17)   # eclipse temurin  → maven:3.9-eclipse-temurin-17
jdk=graalvm(21)   # graalvm          → gradle:8.14-jdk21-graal
```

### Custom builders

For projects that need a non-standard build environment, pass a `Builder` with a custom image or Dockerfile:

```python
from localbox.models import Builder, DockerImage

# Pre-built image
Builder(
    docker_image=DockerImage(image="my-registry/my-builder:latest"),
    command="make build",
)

# Custom Dockerfile (path relative to solution root)
Builder(
    docker_image=DockerImage(dockerfile="assets/dockerfiles/special/Dockerfile"),
    command="make build",
)
```

### Build caches

Each builder type maintains a persistent cache volume to speed up subsequent builds:

| Builder | Cache location (host) | Container path |
|---------|----------------------|----------------|
| Maven | `.build/maven/` | `/var/maven/.m2` |
| Gradle | `.build/gradle/` | `/var/gradle` |
| Gradle (Maven repo) | `.build/maven/` | `/var/maven/.m2` |
| Node.js | `.build/node/` | `/home/node/.npm` |

## Services

A **Service** is a Docker container declared in the solution and managed by Docker Compose. Localbox generates the `docker-compose.yml`; actual container lifecycle (start, stop, logs) is handled by `docker compose` directly.

```python
from localbox.models import Service, DockerImage, ComposeConfig, named_volume, PgCheck

db = Service(
    name="db:primary",
    image=DockerImage(image="postgres:16"),      # pull this image
    compose=ComposeConfig(
        order=1,                                 # start order (lower = earlier)
        ports=["5432:5432"],
        environment={
            "POSTGRES_DB":       "myapp",
            "POSTGRES_USER":     "myapp",
            "POSTGRES_PASSWORD": config.env.db_pass,  # ${db_pass} reference
        },
        volumes=named_volume("db_data", "/var/lib/postgresql/data"),
        healthcheck=PgCheck(),                   # docker healthcheck
    ),
)
```

### Build contexts

When a service has `project=` set, Localbox passes that project's source directory as a named Docker build context to `docker buildx`. This is how compiled artifacts move from the build stage into the service image — without copying files manually.

In your Dockerfile, reference the project by its **local name** (the part after `:`, or the full name for ungrouped projects):

```dockerfile
# Service for project "backend:api" (local_name = "api")
FROM eclipse-temurin:17-jre
COPY --from=api target/myapp.jar /app.jar
ENTRYPOINT ["java", "-jar", "/app.jar"]
```

```dockerfile
# Service for project "frontend:ui" (local_name = "ui")
FROM nginx:alpine
COPY --from=ui dist/ /usr/share/nginx/html/
```

Use `projects=` to include multiple build contexts in a single service image:

```python
gateway = Service(
    name="gateway",
    project=ui,              # primary build context  → --build-context ui=...
    projects=[admin_ui],     # additional context     → --build-context admin_ui=...
    image=DockerImage(dockerfile="assets/Dockerfile.gateway"),
    compose=ComposeConfig(ports=["80:80"]),
)
```

The Dockerfile can then reference both `--from=ui` and `--from=admin_ui`.

---

### Image sources

A service image can come from three places:

| Scenario | `DockerImage` config | What happens |
|----------|---------------------|--------------|
| Pre-built public/private image | `image="postgres:16"` | pulled and tagged |
| Custom Dockerfile in the solution | `dockerfile="assets/Dockerfile"` | built with `docker buildx` |
| Library service (TomcatService, SpringBootService) | _(implicit)_ | Dockerfile generated automatically |

### Grouping and naming

Service names follow the same `group:local_name` convention as projects:

| Name | Compose service | Container name |
|------|-----------------|----------------|
| `"db"` | `db` | `myapp-db` |
| `"db:primary"` | `db-primary` | `myapp-db-primary` |
| `"be:payments:api"` | `be-payments-api` | `myapp-be-payments-api` |

### Dependencies

Use Service objects (not strings) in `depends_on`:

```python
api = Service(
    name="be:api",
    ...
    compose=ComposeConfig(
        depends_on=[db],    # Service object — localbox resolves to compose service name
    ),
)
```

### Image tagging

Built service images are tagged as:

```
{solution.name}/service/{name-with-slashes}:latest
```

For example, solution `myapp`, service `be:payments:api` → `myapp/service/be/payments/api:latest`.

## Environment variables (BaseEnv)

Use `BaseEnv` for typed, IDE-friendly environment variable definitions. Required fields (with no default) raise an error at `compose generate` time if unset:

```python
import dataclasses
from localbox.models import BaseEnv, SolutionConfig, env_field

@dataclasses.dataclass
class Env(BaseEnv):
    db_host: str = env_field()                    # required field, visible in logs
    db_pass: str = env_field(is_secret=True)      # required, masked in output
    cache_ttl: str = env_field()                  # required

config = SolutionConfig[Env](
    name="myapp",
    env=Env(
        db_host="localhost",
        cache_ttl="3600",
        # db_pass intentionally not set here; set in solution-override.py
    ),
)
```

Reference env values through **instance access** on `config.env`. Every attribute
returns an `EnvRef` — a `str` subclass whose string form is `${FIELD}`. Any
f-string or concatenation produces a Docker Compose variable reference, and the
real value is written once to the `.env` file alongside `docker-compose.yml`.

```python
ComposeConfig(
    ports=[f"{config.env.db_host}:5432"],       # → "${db_host}:5432"
    environment={
        "POSTGRES_HOST":     config.env.db_host,    # → "${db_host}"
        "POSTGRES_PASSWORD": config.env.db_pass,    # → "${db_pass}"
    },
)
```

After `localbox compose generate`, the resulting files look like:

```yaml
# docker-compose.yml
services:
  db:
    ports:
      - ${db_host}:5432
    environment:
      POSTGRES_HOST:     "${db_host}"
      POSTGRES_PASSWORD: "${db_pass}"
```

```env
# .env  (generated, gitignored)
db_host="localhost"
db_pass="s3cr3t"
```

This means the committed `docker-compose.yml` stays free of secrets and
machine-specific values; per-environment overrides happen by editing `.env`.

If you need the raw value in Python (rare — mostly for scripting), call
`config.env.raw_value("db_host")`. Referencing a required field that was never
set raises `ValueError` at `compose generate` time, naming the field.

## Library services

The `localbox.library` package contains higher-level service abstractions for common Java deployment patterns. They generate Dockerfiles automatically, so you don't need to write them by hand.

### SpringBootService

For Spring Boot JAR deployments. Auto-detects the artifact, uses the project's JDK to select the runtime image, and adds a Spring Actuator healthcheck:

```python
from localbox.library import SpringBootService

api = SpringBootService(
    name="be:api",
    artifact=api_project.artifact(),
    compose=ComposeConfig(ports=["8080:8080"], depends_on=[db]),
)
```

### TomcatService

For WAR deployments to Tomcat. Supports multiple webapps in one container:

```python
from localbox.library import TomcatService

portal = TomcatService(
    name="be:portal",
    webapps={"portal": portal_project.artifact()},
    tomcat_version="9-jdk17",
    compose=ComposeConfig(ports=["8080:8080"]),
)
```

## Plugin system

Third-party tools can add CLI commands to Localbox by registering a `localbox.commands` entry point:

```toml
# plugin's pyproject.toml
[project.entry-points."localbox.commands"]
quarkus = "localbox_quarkus.cli:quarkus_cmd"
```

The registered value must be a Click command. It appears as `localbox quarkus ...` after installing the plugin. Broken plugins print a warning but do not prevent Localbox from starting.
